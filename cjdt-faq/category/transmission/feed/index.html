]]> Is the CJDT reporting a separate stream from OBTS? https://deseanlewis.github.io/cjdtfaq/cjdtfaq/is-the-cjdt-reporting-a-separate-stream-from-obts Thu, 30 Jul 2020 20:26:25 +0000 https://deseanlewis.github.io/cjdtfaq/?post_type=epkb_post_type_1&p=146 As discussed with the clerks, the future plan is for CCH/OBTS and CJDT to be submitted in one XML transmission. To that end, FDLE is working to define the reporting elements for both reports ensuring that the data requirements do not conflict.

Initially however, CJDT will be a separate transmission in XML while CCH/OBTS will continue to be reported in the pipe delimited format. The data dictionaries for each reporting will also be separate as the rules for elements may differ.

For example, bond-related elements are unique to CJDT and are not in the CCH/OBTS Dictionary. Addtionally, OBTS Dictionary allows for final action values that are not valid in CJDT since CJDT does not contain juvenile information.

]]> Can annual data elements for jail administrators be sent in a separate data feed from monthly, weekly and daily reporting elements? https://deseanlewis.github.io/cjdtfaq/cjdtfaq/can-annual-data-elements-for-jail-administrators-be-sent-in-a-separate-data-feed-from-monthly-weekly-and-daily-reporting-elements Thu, 30 Jul 2020 20:15:20 +0000 https://deseanlewis.github.io/cjdtfaq/?post_type=epkb_post_type_1&p=134 In short, yes, annual data elements can be sent in a separate report from other elements and are only expected once annually.

To elaborate, based on the IEPD design, elements can be segmented even further based on availability. The full details are available in the CJAdmin Information Exchange Package Documentation (IEPD). The Master Document identifies the minimum property sets (i.e. required data elements) as well has how data can be sent. The County Corrections Annual Report message has a cardinality of 0-1 for most elements. That means that a “report” can be submitted with the data available at the time. Elements with nulls are not reported and should instead be omitted from the generated XML report provided they are not required.

For example, if by August 1, the June 30 officers and inmate data is not available, it should be omitted from the August report and sent at a later date. If on September 1, those elements are available they can be transmitted using a Delete/Insert report transaction code.

The samples in the IEPD documentation show all elements associated with a report. The cardinality identifies the data that can be segmented and later sent using the Delete/Insert transaction code.

]]> After the initial data extraction, what triggers a case to be reported? https://deseanlewis.github.io/cjdtfaq/cjdtfaq/after-the-initial-data-extraction-what-triggers-a-case-to-be-reported Thu, 30 Jul 2020 20:11:59 +0000 https://deseanlewis.github.io/cjdtfaq/?post_type=epkb_post_type_1&p=130 Today, the clerks send data (for OBTS/CCH) when there are changes to the reporting elements. Their case management systems identify, through various criteria, if a record is eligible for reporting. For example, is it the right case type, are the minimum required elements available, etc. The evaluation includes new records as well as modified records. Some systems use changes in system dates to make this determination. Others use more complex logic and some require user action. The result is that “active” cases are reported through their lifecycle. Since the future goal is for Clerks to use a single XML submission stream to report CJDT as well as OBTS/CCH data, this type of selection criteria may continue.

For other contributors, the recommendation is for agencies to make trigger determinations based on their workflows. For example with State Attorneys, the decision points at case initiation, filing and closure may be reasonable points in time.

Recommendations for reporting triggers are welcome.

]]> For data elements that are recorded daily, does FDLE expect transmission of these elements everyday or aggregated and sent monthly? https://deseanlewis.github.io/cjdtfaq/cjdtfaq/for-data-elements-that-are-recorded-daily-does-fdle-expect-transmission-of-these-elements-everyday-or-aggregated-and-sent-monthly Fri, 12 Jun 2020 19:25:07 +0000 https://deseanlewis.github.io/cjdtfaq/?post_type=epkb_post_type_1&p=54 Data submission timing is flexible allowing agencies to have the opportunity to send data in near-real time, daily, monthly, etc. This flexibility will allow agencies and FDLE to consider technical issues like file generation complexity and network traffic based on frequency.
